ArgoCD GitOps health — audited + acted 2026-06-24 (worktree exciting-cori).

**app-of-apps:** `nixidy/system/cicd/app-of-apps.nix` (PR #24) generates `Application-apps.yaml` — a self-managing `apps` Application watching `generated/prod/apps/` with selfHeal+prune. It existed in git but was NEVER applied to the cluster, so new git apps never auto-created (a `git merge` alone doesn't deploy a new app). Seeded 2026-06-24 via `kubectl apply -f Application-apps.yaml`; cluster went **46 → 115 Applications** (114 git + the adguard orphan). To bootstrap on a fresh cluster: apply that file once (or `nixidy bootstrap`); it self-heals thereafter. The 68 newly-created apps then deploy their workloads (each child has its own selfHeal+prune).

**Pervasive OutOfSync is COSMETIC, not divergence:** the ~22 long-standing OutOfSync apps drift ONLY on `ExternalSecret` + `HTTPRoute` (and 2 cilium `Gateway`s). nixidy renders minimal manifests; the external-secrets controller / Gateway-API apiserver default extra fields (refreshInterval, conversionStrategy, creationPolicy, deletionPolicy, template.engineVersion, mergePolicy; parentRefs.group/kind, backendRefs.group/kind/weight, implicit matches PathPrefix /). `ServerSideApply` never strips server-managed fields → diff never closes → selfHeal retries 6× then backs off ("Already attempted sync"). Resources are functional (verified ExternalSecret Ready/"secret synced", bitwarden ClusterSecretStore Valid/Ready). FIX = `ignoreDifferences` for those two kinds on the nixidy Application template (one declarative change covers all apps). Do NOT blanket-sync expecting it to clear — it won't.

**adguard:** live Application whose source path `generated/prod/adguard` does NOT exist in git → `Unknown/Degraded` orphan (cloud-burst leftover). It has no finalizer and no `app.kubernetes.io/instance` label, so the app-of-apps will NOT prune it and a prune wouldn't cascade-delete its workloads. Decide: re-home into main or delete the Application.

**INCIDENT 2026-06-24 — seeding all 68 apps at once overloaded sirver (self-recovered):** right after the app-of-apps seed, the cascade burst (68 Applications created + ArgoCD reconciliation + RBD PVC demand from new apps) transiently overloaded **sirver specifically**: rke2-server restarted (NRestarts 256→257), orphaned ~7400 containerd-shims, 1-min load spiked to ~327, and sirver's LOCAL apiserver timed out for ~8 min (even `/healthz`, `get nodes`). octopus+dingo held etcd quorum the entire time — the cluster API via octopus stayed `/healthz=ok`, all nodes Ready, and the app-of-apps reached Synced/Healthy. sirver SELF-RECOVERED in ~8 min (load→5, tasks→1208, local apiserver→ok); NO manual drain/reboot, no data or quorum loss. This was NOT the RBD-cephx wedge from [[rke2-etcd-pipe-freeze-outage]] (CPU stayed ~79% idle, ~1% iowait, no `-13`) — it was process-churn from the rke2-server flap under the write/log burst (same family as the pipe-freeze, localized to one member). LESSON: do NOT seed all apps at once on this etcd-fragile cluster — stage bulk rollouts (sync apps in batches, or seed the app-of-apps with children's autoSync paused, then enable in waves). When kubectl times out, suspect you're hitting the local apiserver of the wedged node — re-point at octopus/dingo (100.64.0.3 / .6 via the hyena `-J root@178.104.61.137` jump) before concluding the cluster is down.
