**Status (2026-06-23): the "internal cluster services + ad-blocking, over tailscale" model is fully deployed server-side** (PR #52; the earlier #51 AdGuard-bind fix was folded in and closed). Remaining is client-side: the user joins laptop/PC/mobile to the headscale tailnet (`tailscale up --login-server=https://headscale.trdos.me`). The off-LAN datapath to internal services still needs a real off-LAN client to confirm.

**What is live:**
- **hyena headscale pushes AdGuard as the tailnet resolver:** `dns = { magic_dns = false; override_local_dns = true; nameservers.global = ["100.64.0.1"]; }`. magic_dns=false ⇒ NO base_domain search domain (that leak is what hijacked pod DNS originally; keep it false).
- **hyena AdGuard** healthy on **`100.64.0.1:53`** (the mesh IP, not 0.0.0.0 — 0.0.0.0 collides with systemd-resolved's `127.0.0.53/54` stub, which is hyena's OWN resolver). `boot.kernel.sysctl."net.ipv4.ip_nonlocal_bind"=1` lets AdGuard bind the tailnet IP before tailscaled assigns it at boot. Config uses the seed pattern (static `adguardhome` user owns `/var/lib/AdGuardHome`; sops renders a SEED; preStart copies it to a writable owned file only if none exists). API on 0.0.0.0:3000.
- **Cluster nodes opt OUT** of the pushed resolver: `--accept-dns=false`. **NixOS gotcha:** `extraUpFlags` only apply on the FIRST `tailscale up` (the autoconnect unit exits early when already Running), so a post-autoconnect oneshot in tailnet.nix re-asserts `tailscale set --accept-dns=false` (verify with `tailscale debug prefs | grep CorpDNS` → false). Without the guard, nodes + CoreDNS-upstream would resolve through hyena over the tailnet (availability coupling) and re-leak a search domain into pods.
- **Off-LAN reach to internal services — NOW VIA A HOST RELAY ON BONOBO (PR #64, 2026-06-25), not the subnet route.** The internal gateway gets a *native* tailnet IP: a `systemd-socket-proxyd` on bonobo (`static/tailnet.nix` `gatewayRelay = "100.64.0.4"`, set in flake.nix) binds bonobo's tailnet IP `100.64.0.4:80/:443` (FreeBind) and forwards raw TCP to the gateway VIP `192.168.50.241` (TLS/SNI pass through → envoy still Host-routes). external-dns-internal now writes `*.trdos.me → 100.64.0.4` into hyena's AdGuard. Verified end-to-end: octopus resolves via AdGuard → 100.64.0.4 → relay → gateway returns byte-identical responses to hitting .241 directly. WHY a host relay (not a pod/tailscale node): headscale denies `tailscale serve`, and Cilium's eBPF L7LB tproxy refuses in-cluster pod→gateway connections. The old `192.168.50.241/32` subnet route is STILL advertised by tailnet.nix but is now **vestigial** for tailnet clients (nothing resolves to .241); drop it in a future all-node deploy. Tradeoff: single relay node — if bonobo is down, internal services are unreachable over the tailnet until DNS is repointed or a second relay added. See [[project_internal_gateway_tailnet_relay]].

**Durable facts (load-bearing for any future tailnet/routing work):**
- **Cilium runs its OWN pod pool: `ipv4NativeRoutingCIDR = 10.0.0.0/8`** (cilium.nix:97), /24 per node (10.0.X.0/24; sirver 10.0.0, octopus 10.0.1, dingo 10.0.2, bonobo 10.0.3, chinchilla 10.0.4). The k8s `Node.podCIDR` (10.42.X.0/24, RKE2 default) is **vestigial** — Cilium ignores it. So pod CIDRs, advertised routes, and the headscale autoApprover all key on **10.0.0.0/8, NOT 10.42**. `kubeProxyReplacement=true` (services `10.43.0.0/16` are eBPF-handled, never hit the route table; ClusterIP base 10.43.0.10).
- **headscale autoApprover** (hyena.nix policy): `autoApprovers.routes = { "10.0.0.0/8" = ["tag:cluster" "default@"]; "192.168.50.241/32" = ["tag:cluster" "default@"]; }`. Cluster nodes register as user **`default` (UNTAGGED** — the `tag:cluster` the cloud-burst plan assumed was never applied), so the approver MUST include `default@`; future CAPI workers register tagged. Quirk: `headscale nodes list-routes` "Approved" column can show blank for an auto-approved route even though it IS approved — the `applying route approval results newApprovedRoutes=[...]` log line + a "Serving (Primary)" entry are authoritative.
- **hyena IS deployable.** The old "MBR+ext4 golden image, undeployable until disko-image alignment" gate comment was STALE (now corrected). Live hyena = GPT+ESP+ZFS matching vps.nix disko; the flake config generates the identical fileSystems+bootloader; deploy-rs switches are reboot-safe. The sops age key is present.
- **Deploy mechanics from a remote Mac** (not on the home LAN, not on hyena's headscale tailnet — so LAN node names like `chinchilla` do NOT resolve, and 100.64.0.x is not directly routable):
  - Cluster nodes: `direnv exec . deploy --skip-checks --hostname <tailnet-IP> --ssh-opts "-J root@178.104.61.137 -o StrictHostKeyChecking=accept-new -o ConnectTimeout=15" '.#<node>'` — ProxyJump through hyena over the tailnet; ssh_user=tristan, passwordless sudo. IPs: sirver 100.64.0.2 / octopus .3 / bonobo .4 / chinchilla .5 / dingo .6. deploy-rs `--hostname` occasionally fails to apply (shows `hostname = "chinchilla"`); just retry.
  - hyena: `deploy --skip-checks --hostname 178.104.61.137 --ssh-user root --ssh-opts "..." '.#hyena'` (direct public IP, root). `hyena` is reachable at 178.104.61.137 (= headscale.trdos.me, direct A record).
  - All these are `switch`es (no reboot), so OSD/etcd reboot hazards don't apply. Production reads/writes on hyena and the nodes are gated by the auto-mode classifier — need explicit user authorization naming the host.
