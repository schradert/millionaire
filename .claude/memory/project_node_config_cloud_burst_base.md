That combination is in no committed branch:
- `main` = May 23 nixpkgs but **no tailnet** → deploying main to a node **rips tailscale off it** (`nix store diff-closures` shows `tailscale 1.98.2 → ∅`).
- `cloud-burst/*` (00-hyena … 06-autoscaler) = has tailnet but pins **Apr 18 nixpkgs** (`b12141e`).

The nodes were deployed from a **local rebase of cloud-burst onto current main** that was never committed/pushed — that exact tree is gone. So **any node-side (NixOS/static/) change must be based on the cloud-burst node config rebased onto current main**, NOT on plain main, or it regresses the tailnet (and triggers a spurious dbus→broker migration when nixpkgs differs).

The cloud-burst work lives in the **`cloud-burst/*` branch series**; the active session/worktree is **`claude/agitated-hoover-6662b5`** on `cloud-burst/06-autoscaler` (`.claude/worktrees/agitated-hoover-6662b5`). Node-side fixes (the [[coredns-magicdns-search-hijack]] kubelet resolv-conf fix, the multus `cni-dhcp` daemon) belong there, not on main.

How to verify a base is correct before deploying: build a node's toplevel and `nix store diff-closures /run/current-system <new>` ON that node — the delta should be only your intended change (e.g. `etc-rke2-resolv.conf`), with NO `tailscale → ∅` and no big nixpkgs/dbus delta.
